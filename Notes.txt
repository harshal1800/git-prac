### **Git & GitHub: Simple Guide**  

### **What is Git?**  
Git is a version control system that save and track changes in code, 
helping developers collaborate and manage projects efficiently.  

### **What is GitHub?**  
GitHub is a cloud-based platform that hosts Git repositories, enabling collaboration, version control, 
and project management.  

---

### **Key Git Concepts:**  

- **Repository (Repo):** A storage location for code and its history.  

- **Commit:** Saving changes in Git.  
- **Push:** Sending local commits to a remote repository (e.g., GitHub).  
- **Pull:** Fetching and merging updates from a remote repository into your local system.  

- **Branch:** A separate version of the project for new changes.  
- **Merge:** Combining branches.  

- **`git checkout -b <branch-name>`:** Creates and switches to a new branch in one step.  
- **`git checkout <branch-name>`:** Switches to an existing branch in Git.  

- **Clone:** Copying a repository from GitHub to your local system.  
- **Fork:** Creating a copy of someone’s repository in your GitHub account.  
- **Staging Area:** A space where changes are prepared before committing.  
- **Remote Repository:** A Git repository stored on a server (like GitHub). 

- **Fetch:** git fetch is used to update your local repository with the latest changes from the remote repository without merging them.
- **Rebase:** Moving or combining commits in a different branch.  
- **Reset:** Undoing changes by moving back to a previous commit.  

- **Cherry-Pick:** Applying specific commits from one branch to another.  
- **`.gitignore` (Gitignore):** A file that tells Git which files to ignore (e.g., logs, environment files). 

- **`ls` :** Lists all visible files and directories in the current folder.  
- **`ls -a` :** Lists all files, including hidden ones (files starting with `.` like `.git`).  

---

### **Key GitHub Features:**  
- **Issues:** Used to track bugs or feature requests.  
- **Pull Requests (PR):** A way to propose changes before merging them.  
- **Actions:** Automate workflows, like testing or deployment.  
- **Wiki:** Documentation space inside a GitHub repository.  
- **Projects:** A Kanban-style board for task management.

# Git Workflow (Systematic Order)

# 1. Clone a Repository (If You Haven’t Already)
git clone https://github.com/your-username/repository-name.git

# 2. Navigate to the Project Directory
cd repository-name

# 3. Check the Status of Your Repository
git status

# 4. Stage Changes (Move Files to Staging Area)
# To stage all changes
git add .
# To stage a specific file
git add filename

# 5. Commit Changes (Save Them Locally)
git commit -m "Your commit message"

# 6. Pull the Latest Changes from Remote (If Needed)
git pull origin main  # Use 'main' or 'master' depending on your branch

# 7. Push Changes to the Remote Repository
git push origin main  # Use 'main' or 'master' depending on your branch

# 8. View Git History
git log

The `git reset` command has three main modes: **soft, mixed, and hard**, each affecting the commit history, staging area, and working directory differently.

### **1. Soft Reset (`--soft`)**  
```bash
git reset --soft HEAD~1
```
- **Moves HEAD** back to the previous commit.
- **Keeps changes in the staging area** (ready to commit again).
- **Does not affect the working directory** (your files remain unchanged).

**Use case:** When you want to modify the last commit but keep the changes staged.

---

### **2. Mixed Reset (`--mixed`)** *(Default)*
```bash
git reset --mixed HEAD~1
```
- **Moves HEAD** back to the previous commit.
- **Unstages changes** (moves them from the staging area back to the working directory).
- **Does not delete the actual changes** from your files.

**Use case:** When you want to edit changes before committing again.

---

### **3. Hard Reset (`--hard`)**  
```bash
git reset --hard HEAD~1
```
- **Moves HEAD** back to the previous commit.
- **Deletes changes** from the staging area and the working directory permanently.

⚠ **Warning:** This removes all uncommitted changes. Use carefully!

**Use case:** When you want to completely discard changes.

---

### **Summary Table**
| Reset Type | Moves HEAD | Unstages Changes | Deletes Changes from Working Directory|
|------------|----------- |------------------|---------------------------------------|
| `--soft`   | ✅ Yes    | ❌ No            | ❌ No                                 |
| `--mixed`  | ✅ Yes    | ✅ Yes           | ❌ No                                 |
| `--hard`   | ✅ Yes    | ✅ Yes           | ✅ Yes (Permanent)                    |

### **Unstaging a File and Committing Another in Git**  

When you stage both `file1.txt` and `file2.txt` using `git add`, they move to the **staging area**. 
If you run `git reset file2.txt`, only `file2.txt` is removed from the staging area, but its changes remain in the **working directory**. 
Meanwhile, `file1.txt` stays staged.  

When you commit using `git commit -m "Committed file1"`, only `file1.txt` gets committed, while `file2.txt` remains uncommitted 
and can be modified or staged later.

### **Add a Local Project to a Remote GitHub Repository**  

Follow these steps to push your local project to GitHub:  
# 1. Navigate to your project folder
cd your-project-folder  

# 2. Initialize Git (if not already initialized)
git init  

# 3. Add all files to the staging area
git add .  

# 4. Commit the changes
git commit -m "Initial commit"  

# 5. Add the remote repository (replace with your repo URL)
git remote add origin https://github.com/your-username/your-repo.git  

# 6. Push the project to GitHub
git push -u origin main  # Use 'main' or 'master' depending on your branch

### **Branching and Merging in Git**  

#### **1️⃣ Create a New Branch (Branching)**  

git branch branch-name   # Create a new branch
git checkout branch-name # Switch to the new branch
# OR (shortcut)
git checkout -b branch-name
```

#### **2️⃣ View All Branches**  
```sh
git branch
```

#### **3️⃣ Merge a Branch into `main` (Merging)**  
```sh
git checkout main       # Switch to main branch
git merge branch-name   # Merge the branch into main
```

#### **4️⃣ Delete a Branch**  
```sh
git branch -d branch-name  # Delete the branch (if merged)
git branch -D branch-name  # Force delete (even if not merged)
```

### **Git Merge Conflict (Theory)**  

A **merge conflict** occurs when Git is unable to automatically merge changes from two branches 
because the same part of a file has been modified differently in both branches.  

#### **Causes of Merge Conflicts**  
1. Two people change the same line in a file on different branches.  
2. A file is deleted in one branch but modified in another.  
3. Conflicting changes exist in multiple commits.  

#### **How to Resolve a Merge Conflict?**  
1. **Identify conflicts** using `git status`.  
2. **Manually edit the conflicted files** to remove merge markers (`<<<<<<<`, `=======`, `>>>>>>>`).  
3. **Stage the resolved files** using `git add filename`.  
4. **Commit the merge** with `git commit -m "Resolved merge conflict"`.  
5. **Push the changes** to update the remote repository.
